## Текст поста

Если вы разрабатываете продукт с русским интерфейсом, но пишете код на английском, вы уже сталкивались с ошибками, которые обычные линтеры не видят: сломанная плюрализация в локалях, `"Добавить в " + корзина` в шаблонах, идентификатор `kolichestvo_tovarov`, половина комментариев в файле на русском — вторая половина на английском.

LLM-ассистенты (включая Claude) такие вещи регулярно пропускают. Я собрал скил для Claude Code, который их ловит.

## Что проверяет

**1. Структура плюральных форм в i18n** — `i18next`, `ICU MessageFormat`, `vue-i18n`. По правилам CLDR в русском четыре формы: `one`, `few`, `many`, `other`. Если в `ru.json` есть только `item_one` и `item_other` — скил это ловит. Если ICU-сообщение выглядит как `{count, plural, other {# товаров}}` — это синтаксически валидный ICU, но грамматически сломанный русский: для 1, 2, 3, 4 он даст неправильную форму. Встроенный линтер FormatJS такую проблему не проверяет.

Ещё одна частая ошибка, которую ловит скил: значение ключа содержит `{{count}}`, а плюральных сиблингов у ключа нет. `"У вас {{count}} новых уведомлений"` — классический баг, который выдаёт `"1 новых уведомлений"`.

**2. Согласованность терминологии.** Если в одном файле `пользователь`, а в соседнем — `юзер`, скил это найдёт. Можно переопределить автоматику через опциональный `.claude/russian-glossary.yml`, где явно зафиксировать канонический термин.

**3. Согласование падежей в склеенных строках.** `` `Удалить ${item.name}?` `` — глагол «удалить» управляет винительным падежом, а `item.name` вставляется в именительном. Для большей части русской лексики это даёт неграмматичный текст. Детектор эвристический (закрытый список предлогов и переходных глаголов), но срабатывает на типовые случаи. Severity — `Warning`, никогда не `Error`, потому что эвристика не идеальна.

**4. Смешанные языки в одном файле.** Файл, где половина комментариев на русском, половина — на английском, плюс отдельная проверка на русские строковые литералы вне i18n-системы. Пороги документированы: меньшинство должно составлять минимум 25% от всех значимых комментариев, и минимум два комментария.

**5. Транслитерация в идентификаторах.** `polzovatel_tovarov`, `kolichestvo`, `nazvanie`. Детектор трёхуровневый: нужно словарное совпадение плюс либо фонотактический признак (`shch`, `zh` после согласной, `kh` перед гласной), либо подтверждённое наличие кириллицы в файле. Severity — `Info`, никогда не выше: эвристика низкой уверенности, публичных доказательств у этой категории меньше всего (проблема живёт в приватных корпоративных кодовых базах).

## Чего НЕ делает

Типографику («ёлочки», тире, неразрывные пробелы), орфографию, ё/е — это уже хорошо покрывают другие инструменты, и дублировать их я не хотел:

- **Типографика и UX-тексты** → [talkstream/ru-text](https://github.com/talkstream/ru-text) — другой скил для Claude Code, около 1040 правил по прозе. Ставьте рядом, они не конфликтуют и дополняют друг друга.
- **Орфография** → [yaspeller](https://github.com/hcodes/yaspeller)
- **Буква ё** → [eyo](https://github.com/e2yo/eyo)

## Ноль шума в английских проектах

Скил активируется только если в проекте действительно есть русский: минимум 10 символов кириллицы в коде и локалях плюс хотя бы одна последовательность из пяти подряд идущих кириллических символов. Для чистых англоязычных репозиториев он просто молчит. `node_modules`, `dist`, `build` и пути из `.gitignore` не сканируются.

## Установка

```bash
git clone https://github.com/Anic888/russian-text-quality.git
ln -s $(pwd)/russian-text-quality ~/.claude/skills/russian-text-quality
```

Или `cp -r russian-text-quality ~/.claude/skills/`. Claude Code подхватит скил автоматически.

Проверить: в проекте с русским контентом попросите Claude Code «проверь проект на качество русского текста». В ответ придёт отчёт в формате Markdown с разделами по severity — errors, warnings, info — и ссылками на конкретные строки.

## Ограничения, о которых я говорю честно

- `i18next` v3 (`key_plural`) не поддерживается — только v4.
- Эвристика для `vue-i18n` считает количество `|`-сегментов. Проза с литеральным `|` может вызвать ложное срабатывание, поэтому такие находки помечаются `Warning`, а не `Error`.
- Проверка #3 (склейка) — регулярки по закрытому списку. Ожидаемый процент ложных срабатываний около 10–15%.
- Проверка #5 (транслитерация) — самая слабая по публичным доказательствам: в open-source такие идентификаторы встречаются редко, основная масса — в приватном коде. Поэтому severity `Info`, не `Warning`.
- Вложенные ICU-плюрали (plural внутри select) v1 может пропустить — регулярный парсер не обходит AST полностью.

## Как помочь

Расширить словарь транслитерации, добавить триггеры для проверки склейки, поделиться реальными примерами — всё через PR. Правила — в `CONTRIBUTING.md`. Короткая версия: на любое утверждение о русском нужна ссылка на авторитетный источник (gramota.ru, Розенталь, CLDR), а на каждый пример — ссылка на реальный код или публикацию. Выдуманных примеров в репозитории нет и не будет.
